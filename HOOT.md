# WISDOM.md

## Current Plan

Publish and maintain Owlbox as a repository-local continuity system for long-lived projects.

An Owlbox contains the three OWL files. Owlbox-enabled adds generated `HOOT.md` and `scripts/hoot-hoot.ps1`. Owlbox-enabled, assisted adds the four Owlbox skills.

This repository provides the specification, templates, scripts, distributable skills, and its own Owlbox for maintaining the Owlbox tool project.

Current work is the staged pre-handoff diet review of Owlbox, Ledger, and Lulu, with every repository change gated by explicit user approval.

## Preferences

- `HOOT.md` is already reserved by the Owlbox system as the generated continuity view. Do not reuse it for Owlbox publication/spec content.
- `VISION.md` is the generated concatenation of Owlbox repository project files: a vision about the plan, not the repo continuity layer.
- `SOP.md`, `FILES.md`, `TEMPLATES.md`, `BUILDABOX.md`, `assistant/`, and `scripts/` belong to the Owlbox repository's publishing and integration support surface.
- Owlbox deliverables, destinations, enablement/build, and verification are documented in `DELIVERABLES.md`.
- Installed Owlbox deliverables create one Owlbox for the containing project. Nested Owlboxes are for nested repository scopes, not for the installed Owlbox file set itself.
- An Owlbox is the `owlbox/` folder containing the three OWL files.
- An Owlbox-enabled project has the supported implementation: the `owlbox/` folder, the three OWL files, generated `HOOT.md`, and `scripts/hoot-hoot.ps1`.
- An Owlbox-enabled, assisted project also has the four Owlbox assistant skills available.
- "OWL files" refers only to `owlbox/OUTLINE.md`, `owlbox/WISDOM.md`, and `owlbox/LEGACY.md`. `HOOT.md` is the generated combined view of the OWL files and is an Owlbox deliverable, but it is not itself an OWL file.
- The Owlbox repository's own `HOOT.md` and `owlbox/` files are project continuity for maintaining the Owlbox tool project.
- `LEGACY.md` Log is historical provenance and disaster recovery material: the sacrosanct heart of the program, but not an ordinary source of truth for current plan, state, or preferences. Use Log to reconstruct state only when the rest of the Owlbox system has failed or been lost.
- `LEGACY.md` explicitly accepts entire handoff files when full context transfer matters. After adding a handoff file, evaluate it and place relevant active or durable information in `OUTLINE.md` Plan, `OUTLINE.md` Journal, `WISDOM.md` Current Plan, and `WISDOM.md` Preferences as needed. If the handoff does not affect those sections now, preserve it only in `LEGACY.md`.
- Owlbox transfer moves project continuity by transferring the three OWL files. A project-window handoff package should also include a comprehensive chat-history and project-status summary.
- The Owlbox specification must be capable of reproducing the assistant Owlbox skills exactly. Do not paraphrase skill or template language during copy or publication work unless an intentional spec change requires it.
- The assistant Owlbox implementation is a four-skill package: `owlbox`, `owlbox-outline`, `owlbox-wisdom`, and `owlbox-legacy`.
- The file-specific Owlbox skills should protect the user's will by enforcing the permanent OWL file headers without introducing a generic blocker workflow. They should stop and ask only when a requested edit would violate a file's handling rules.
- Repository source skill packages and installed runtime skill packages should match after approved skill changes.
- Major Owlbox repository changes should be recorded as audit/changelog material in the OWL files so future review can distinguish active state, durable direction, and historical change record.
- The correct local repository base path is `C:\Users\mch\Repositories\<name>`.
- Formal files should use "rules" rather than "commandments"; "commandment" is conversational shorthand only.

#### Owlbox Tool Project Changelog Handoff

The Owlbox repository was created to publish Owlbox as a reusable continuity system for Git repositories. Early planning separated Owlbox instances from the support material in this repository. Owlbox instances contain `OWLBOX.md`, `owlbox/OUTLINE.md`, `owlbox/WISDOM.md`, and `owlbox/LEGACY.md`. The rest of this repository exists to explain, reproduce, package, and support that system: project documentation, file descriptions, template documentation, build scripts, templates, and a Codex skill implementation.

The role of `OWLBOX.md` was corrected during design. `OWLBOX.md` is already reserved by the Owlbox system as the generated continuity view for whichever repository contains it. For the Owlbox repository, that means `OWLBOX.md` describes the Owlbox tool project itself. The generated publication/spec view was instead named `VISION.md`, described by the user as "a vision about a plan." `VISION.md` is the generated unified view of the Owlbox system vision and specification, while `OWLBOX.md` remains this repository's own continuity file.

The project adopted a strict preservation rule for the Codex Owlbox skill and templates. The Owlbox repository should be able to reproduce the live Owlbox skill exactly. Copying, paraphrasing, or accidentally weakening skill or template language is treated as Fray. Any intentional change to the skill/template language should be made deliberately and then reflected through the repository support files, rather than introduced accidentally during publication work.

Handoff handling was clarified. `LEGACY.md` may preserve entire handoff files when full context transfer matters. Those handoffs should be evaluated for information that belongs in active or durable Owlbox sections, but the rule is not that every handoff must be forced into Plan, Journal, Current Plan, or Preferences. If a handoff is useful history but does not affect active work or durable project preferences now, preserving it in `LEGACY.md` is enough.

The Owlbox repository's own Owlbox was corrected to describe the history, working memory, preferences, and plan of the Owlbox tool project. This distinction matters because this repository is both an example of Owlbox in use and the source project for the Owlbox tooling. Its continuity files should therefore track the development and maintenance of the tool project.

The Owlbox repository's own Owlbox is for project continuity when maintaining the Owlbox tool project. Adopting repositories receive their own fresh Owlbox instance built from the templates and tooling.
